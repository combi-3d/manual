    CPU

    CPUのコア数と周波数が、RAW現像と、Photoshop自動処理に効きます。Photoshopではhyperthreding (HT)は、効きませんでした（2019年）。いまどきのIntel i5, i7, i9なら、どれでもよく働きます。

    GPU

    DDR5 8GBやHBM2 4GBで十分働きます。GPUは、3Dレンダリングに効きます。規格はDDR4, DDR5, DDR6, HBM2の順で高性能になります。HBM2規格は高性能だがコストがかかる、DDR6規格は高性能かつ安い（2020年)、と評されます。Apple M1は、どうやら、DDR5 4GB (560X)相当らしく、総合的には3Dイメージングの能力がありそうです。

    メモリ

    最低容量は16GBですが余裕はありません。32GBから余裕がでてきます。64GBあれば安心です。

    ImageJで連続画像を処理するとき、容量が大きいほうがいいです。16GBでも画像処理を実行できますが、慎重に行う必要があります。具体的には、連続画像は、メモリに展開せず、virtual stackやmacro機能をつかって、メモリに優しく作業をすすめます。32GB以上あれば、このような気配りはいらなくなり、余裕がでてきます。くわしくは、下記の「ImageJで扱う連続画像」「ImageJに割り当てるメモリ量」をご覧ください。

    ImageJ以外では、RAW−JPEG変換がメモリを要求します。変換中、メモリが開放されないようで、約100枚の変換ごとに1GBのメモリが喰われていきます。メモリオーバーになると、変換が非常にゆっくりになり、「つづきはまた明日」となるでしょう。

    ストレージ

    SSD

    外付けSSDを、CoMBIデータを一時的に保管し、画像処理の作業をする場所として利用しています。理由は、データの読み書きが早いことと、PC本体のSSDを守れるかもという期待があるからです。

    SSDには寿命があります。SSDの寿命は、容量が大きいほど長く、単純に比例します。大容量に投資するメリットとして、保存できるデータ量が増えることと、長寿命であることも挙げられます。外付け512GBSSDをCoMBIで使い続けたところ、4年後にディスクが読めなくなり寿命を迎えました。ゲームでSSDをよく使うひとも同様に、早さの恩恵を受けるとともに、4年程度の寿命だと覚悟をもって512GB SSDを使っているそうです。いつ寿命が来てもいいように、画像処理の作業中は、元データのRAWファイルを複数箇所に確保しています。SSDとSDカード、またはSSDと長期保存用HDDです。

    HDD

    長期保存にHDDを使っています。「RAID対応外付けHDD」をメイン保管場所にしています。CoMBIでRAW撮影する場合、1標本で10〜100GB、年間1〜1.5TBになります。もちろん実験の勢いにもよります。HDDを準備する際は、多めに見積もって、「1標本50GB」「20サンプル/1年/1TB」にしておけば、しばらくもつと思います。データとPC本体のデータは、念の為、MacのTimeMachineでもう1枚のHDDにバックアップしています。

    画像ファイルのサイズ

    連続画像は重いといっても、どれくらいメモリをくうのか、実際はどのくらいのデータサイズになるでしょうか。ファイル形式は、RAW, TIFF, JPEGどれで撮影するのが良いのでしょうか。

    画像形式と一枚のサイズ

    表は、各種画像形式でのカラー画像一枚のサイズです。JPEGは、おおかたのサイズです。圧縮形式なので変動します。もしCoMBIで1000枚撮影すれば、MBをGBと読んでください。保存ディスクの容量を見積もるのに役立ててください。フルサイズ（Full-size）センサーなら、full-sizeで撮影するか、APS−C相当にクロップするかを、選べます。広くも撮れるし、データサイズの節約も使用と思えばできます。RAWは、豊かな階調数をもっているので、元データに最適です。結局、CoMBIの撮影では、RAWで撮影、JPEGで画像処理をしています。

    ImageJで扱う連続画像

    ImageJで連続画像をひらくと、ピクセル数に応じた容量でメモリを喰います。連続JPEG画像なら、圧縮された容量でディスクに保存されていますが、いざ、ImageJでstackとして開くと、圧縮前のサイズでメモリを使用します。表は、様々なサイズと枚数の8 bit画像の合計サイズをしめします。 経験的にも、計算上（下記）も、ImageJで軽快に扱える限界は、ざっと1000x1000x1000=3GBあたりまでと思って、画像処理の計画を立てています。グレースケール画像なら、カラー画像の3分の1の容量になります。

    ImageJに割り当てるメモリ量

    ImageJ/FIJIでは、Edit> Options> Memory & Threads...より、割り当てるメモリサイズを入力します。メモリサイズは、PC搭載メモリの70％を指定するとよいといわれています。下表は、PC搭載メモリとImageJで使用可能なメモリサイズを計算したものです。MacでOSと常駐アプリを7GB、ImageJへの割当を70%としました。結果、最小構成は16GBです。おすすめは32GBで余裕をもたすです。

    ImageJで使用できる最大と、半値を示しています。半値を下回る連続画像であれば、ImageStabilizer処理前と処理後の連続画像を両方開いて比較できます。ImageJでは、メモリを食わないモード、virtual stackがあります。しかし、virtualモードでは、プラグインによっては動作しない場合があるため、やっぱり、メモリは潤沢にあって、データはすべてメモリに展開できたほうが安心です。

    実際には、スワップもあって、常駐部分は少なくてもよさそうですが、連続画像を軽快に扱うためにはメモリは潤沢であるのに越したことはありません。

    3D slicerで扱う連続画像

    3D slicerが推奨する画像データサイズの限界は、「搭載GPUのVRAMの半値」です。例えば、GPU DDR5 8GBなら、4GBが半値となり、おおかたカラー画像1200x1200x1000枚 = 4.3GBに相当します。経験的にも、Horos/OsiriXでも、そのへんが限界です。ただし、グレースケールなら容量は3分の1になります。グレースケール由来のVolume Renderingを行うときは、もうちょっと大きめの、多数の画像をあつかえます。3Dの作業中、表示画質を落とすといった工夫も、必要に応じてやっています。
